Build an AI agent from scratch in ~60 lines of Python

Learn · Unit 1 · July 2026 · No framework required

If you ran the interactive agent loop, you've already seen everything this code does. That's the point: an agent is a small machine, and it's worth building — or at least reading — the small machine once before any tool or framework abstracts it away. This is also what Anthropic's own engineering guidance recommends — start simple, add complexity only when it earns its keep.

What we're building: an agent with two tools (a calculator and a file reader) that can answer questions like "read expenses.txt and tell me the quarterly total." A model API, a while-loop, and a dictionary of tools. That's the whole ingredient list.

The complete program

This uses the Anthropic Python SDK (pip install anthropic, and set ANTHROPIC_API_KEY). Check the current model list and use whatever the current mid-tier model is — model names rotate, the loop is forever. The same structure works with any provider that supports tool use.

# agent.py — a complete agent, no framework
import anthropic

client = anthropic.Anthropic()
MODEL = "<current-model-name>"   # check the provider's model list

# ---- 1. The tools: schemas the model sees, functions we run ----
TOOLS = [
    {
        "name": "calculator",
        "description": "Evaluate an arithmetic expression, e.g. '(45+87.5)*3'. "
                       "Use this for ALL arithmetic instead of computing yourself.",
        "input_schema": {
            "type": "object",
            "properties": {"expression": {"type": "string"}},
            "required": ["expression"],
        },
    },
    {
        "name": "read_file",
        "description": "Read a local text file and return its contents.",
        "input_schema": {
            "type": "object",
            "properties": {"path": {"type": "string"}},
            "required": ["path"],
        },
    },
]

def run_tool(name, args):
    """Plain code. The model never touches the world - this function does."""
    try:
        if name == "calculator":
            # eval() is fine for a demo; use a real parser in production.
            return str(eval(args["expression"], {"__builtins__": {}}, {}))
        if name == "read_file":
            with open(args["path"]) as f:
                return f.read()[:4000]          # truncate: context costs money
        return f"Unknown tool: {name}"
    except Exception as e:
        return f"ERROR: {e}"                    # errors are just observations

# ---- 2. The loop: think, act, observe, repeat ----
def agent(goal, max_turns=10):
    messages = [{"role": "user", "content": goal}]
    for turn in range(max_turns):
        response = client.messages.create(
            model=MODEL,
            max_tokens=1024,
            system="You are a careful assistant. Work step by step. "
                   "Use the calculator for all arithmetic.",
            tools=TOOLS,
            messages=messages,
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":  # no tool call = agent is done
            return "".join(b.text for b in response.content if b.type == "text")

        results = []                            # run every requested tool
        for block in response.content:
            if block.type == "tool_use":
                print(f"  turn {turn+1}: {block.name}({block.input})")
                output = run_tool(block.name, block.input)
                results.append({"type": "tool_result",
                                "tool_use_id": block.id, "content": output})
        messages.append({"role": "user", "content": results})
    return "Stopped: hit max_turns without finishing."

if __name__ == "__main__":
    print(agent("Read expenses.txt and tell me the quarterly total (monthly x 3)."))

Reading the machine

The tool schemas are prompt engineering. The model chooses tools based entirely on the description fields — notice ours explicitly says "use this for ALL arithmetic," because models left to their own devices will confidently do math in their head and get it wrong. Writing good tool descriptions is half of agent quality, which is why Anthropic's guide talks about designing the "agent-computer interface" with the same care as a UI.

The exit condition is an absence. The loop ends when the model replies without requesting a tool (stop_reason != "tool_use"). Nobody tells the agent it's done — deciding that is the agent's job. This is the single line where "autonomy" lives, and it's also why max_turns exists: an agent that can decide to continue can also fail to decide to stop, and the turn cap is your circuit breaker against a very expensive infinite loop.

Errors are observations, not exceptions. run_tool catches everything and returns error text into the conversation. Delete that try/except and a missing file crashes your program; keep it and the model reads "ERROR: no such file," reconsiders, and tries something else. You watched this exact behavior in the interactive with the failure switch on — these five lines are where that resilience comes from.

The history is the memory. The model itself is stateless; messages — growing every turn — is the agent's entire mind, re-sent and re-read on every call. That has a price: each turn costs more tokens than the last. Real systems summarize, truncate, and retrieve context just-in-time, which is Unit 3's whole topic.

Exercises (each one teaches a production concept)

Add a list_files tool and watch the agent chain it with read_file unprompted — that's planning emerging from tool composition. Print token counts from response.usage each turn and watch the cost curve bend upward — that's the context problem. Make run_tool ask you y/n before any file write — that's a human-approval guardrail, the pattern every production system needs for consequential actions. Then break things on purpose: rename expenses.txt and watch recovery happen.

"But nobody writes this by hand anymore"

True. In practice, you'd open Claude Code, Codex, or an agentic IDE, describe the agent you want, and the code would be written for you — probably better than either of us would write it. So why read this page at all? Two reasons, and they're practical, not sentimental.

First: those tools are this exact loop. Claude Code is a model, a set of tools (edit files, run commands, search), and a repeat-until-done cycle — the same three ingredients you just read, wearing a terminal. You didn't just read a toy; you read the blueprint of the tool everyone actually uses.

Second: generated agents still break, and the fixes live at this level. When an agent loops forever, burns through tokens, or confidently calls the wrong tool, "regenerate it" often isn't enough — you need to know which part of the machine is misbehaving: the tool descriptions? the exit condition? the context growing too fat? Reading ~60 lines once is the cheapest way to be able to answer that.

A practical way to have it both ways: paste this whole page into Claude Code and ask it to build and run the agent, then watch what it does. You'll get the working program without typing it, and you'll recognize every move it makes.

So what do frameworks add?

Now you can evaluate them from strength instead of dependence. LangGraph adds durable state, retries, and checkpoints around this loop; CrewAI adds role decomposition on top of it; MCP standardizes how the tools plug in. All of them are this while-loop wearing increasingly nice clothes — and when a framework misbehaves, you'll now recognize which part of the underlying machine is actually failing.

Honesty corner: the demo uses eval() for arithmetic — sandboxed, but still not something to ship. And a real deployment needs cost tracking, logging, and eval before you trust it with anything that matters. Those are upcoming units, not footnotes.
